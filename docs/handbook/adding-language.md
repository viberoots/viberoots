# Adding a Language

This guide outlines how to introduce a new language without changing the core architecture. Follow the same glue pattern used for Go: exporter labels → provider sync → auto-map → macros consume providers.

## Requirements

- Patches live under `patches/<lang>/` (flat). One patch per key (e.g., `package@version`).
- A language-specific sync generator writes `third_party/providers/TARGETS.<lang>.auto` from patches.
- The exporter emits recognizable labels for targets of this language.
- `tools/buck/gen-auto-map.ts` maps those labels to provider names using helpers in `tools/lib/providers.ts`.

## Steps

1. Exporter labels
   - Extend the exporter to attach language-specific labels to relevant targets (e.g., `module:<import>@<version>` or a `lockfile:` style).
   - Keep label strings stable and deterministic; they form the contract to `gen-auto-map.ts`.

2. Provider sync
   - Add a zx script that scans `patches/<lang>/*.patch` and writes `third_party/providers/TARGETS.<lang>.auto` deterministically.
   - Use `tools/lib/providers.ts` for naming helpers so provider names are unique and stable.

3. Auto-map integration
   - Update `tools/buck/gen-auto-map.ts` to translate your labels into provider labels.
   - Ensure per-target provider lists are sorted and deduplicated.

4. Macros consumption
   - Your macros should load `//third_party/providers:auto_map.bzl` and append providers from the mapping, preserving existing deps and order.

5. CI stages and local flow
   - Glue is generated by Node zx scripts (exporter → sync providers → auto-map); glue is not committed.
   - Ensure `tools/dev/install-deps.ts` runs your language’s sync step locally after exporter.

## Notes

- Keep implementation language-agnostic in the planner (`tools/nix/lang-templates.nix`) and add only what’s necessary.
- Node importer-scoped providers are optional/future; do not rely on them here.
  - If you explore Node later: mirror the same flow with a `sync-providers-node` generator and have `gen-auto-map.ts` translate `lockfile:<path>#<importer>` labels via `providerNameForImporter`. Keep it deterministic and idempotent.
