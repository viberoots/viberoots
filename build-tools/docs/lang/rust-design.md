## Rust as a First‑Class Language — Design and Integration Plan

Audience: engineers and LLM agents implementing the Rust path. Scope: add Rust with the same architectural patterns used for Go and Node: Buck2 orchestrates, Nix builds via dynamic derivations, patching is flat and idempotent, providers drive fine‑grained invalidation.

### Goals and philosophy alignment

- Deterministic, minimal, reproducible: Buck computes impact; Nix provides hermetic builds; patching is explicit and traceable.
- Reuse shared glue and naming: prefer existing helpers in `build-tools/tools/lib/providers.ts`, `build-tools/tools/buck/gen-auto-map.ts`, and the provider sync orchestrator.
- Flat patches directory, one patch per crate@version; no vendoring into the repo.
- CI guardrails: dev overrides are forbidden in CI; glue is generated by zx scripts, not committed.

### Linking expectations

I follow the repo-wide linking model described in `build-tools/docs/cpp-linking.md`, `build-tools/docs/wasm-linking.md`, and `docs/history/build-system/linking-roadmap.md`. If this language does not introduce native or cross-language linking, `deps` remains a graph edge list and no link intent is inferred.

- `deps` is the Buck graph edge list. It does not imply link intent.
- `link_deps` declares linkable inputs. `header_deps` is include-only when that concept applies.
- Macros compute `deps := deps ∪ link_deps ∪ header_deps` deterministically and validate `link_closure_overrides` keys.
- `link_closure` defaults to `"direct"`. `"transitive"` follows `link_deps` only via `build-tools/tools/nix/planner/link-closure.nix`.
- Ordering is deterministic and unsupported deps fail fast with targeted errors.

### C interop requirement

If the language can support C interop, I must provide a documented and tested path to link or call C code using the repo linking model (explicit `link_deps` and deterministic closure). If the language cannot support C interop, this doc must state why and list the constraints.

### Shared wiring and contracts (current repo)

Use the canonical helper surface from `//build-tools/lang:defs_common.bzl` and `//build-tools/lang:language_wiring.bzl`. Macro call sites should not re‑implement wiring or load provider maps directly.

- Preferred macro entrypoint: `prepare_language_wiring(...)` (non‑mutating), with `wiring=` for `genrule`, `nix_calling_genrule`, `non_genrule`, or `srcsless_rule`.
- Provider wiring: load `MODULE_PROVIDERS` from `//build-tools/lang:auto_map.bzl` and use `providers_for`/`realize_provider_edges` for deterministic provider edges.
- Lockfile labels (importer‑scoped languages): `lockfile:<path>#<importer>` with supported importer roots `.` and `projects/apps/*`/`projects/libs/*`; importer‑scoped macros must live in the importer package so importer‑local patch globs are valid action inputs.
- Patch model contract: `build-tools/lang/lang_contracts.bzl` and `build-tools/tools/lib/lang-contracts.ts` define `patch_scope:*` stamping and whether glue runs on patch apply/remove.
- Global Nix inputs: for Nix‑calling macros, use `wire_global_nix_inputs(...)` so `global_nix_inputs()` are real action inputs; labels are observability only.

### Build route policy

Policy for this language:

- Implement artifact-producing macros as Nix-backed builds.
- Keep Buck as graph and test-impact orchestrator, not the producer of production artifacts.
- Allow orchestration wrappers that call `nix build` when inputs remain hermetic and deterministic.
- Allow probe-only non-build macros only when explicitly documented as non-artifact paths.
- Do not introduce fallback Buck artifact build paths for convenience.

### Runnable contract alignment

Rust target run behavior is expressed through manifest runnable contracts:

- Binary/app outcomes publish `run.prod`.
- `run.dev` is optional; Rust defaults to production-style execution unless a dev command is declared.
- `kind:lib` targets remain non-runnable and are excluded from runnable-target summaries.

### Enforcement integration requirement

Language rollout is not complete if it only adds build plumbing. I also need to keep migration
policy enforcement current:

- Add and maintain public macro rows in `docs/handbook/nix-gaps.md`.
- Keep intentional non-build macros in `docs/handbook/nix-gaps-exceptions.json` with
  `kind = "probe-only"` and non-empty justification.
- Extend `build-tools/tools/dev/nix-gaps-inventory-check.ts` and related tests under
  `build-tools/tools/tests/dev/` when route contracts change.
- Ensure required repo validation runs this checker so doc/policy drift fails before merge.

### Path invariants (must‑follow)

- Patches live at `patches/rust/*.patch` (flat; no subdirectories).
- Nix language templates extend `build-tools/tools/nix/lang-templates.nix` via `build-tools/tools/nix/templates/rust.nix`.
- Buck macros live under `build-tools/rust/defs.bzl` and append provider deps using `MODULE_PROVIDERS` loaded via `//build-tools/lang:auto_map.bzl` (do not load `//third_party/providers:auto_map.bzl` directly).
- Provider rules are generated under `//third_party/providers/*` (Rust file: `TARGETS.rust.auto`).
- Exported graph lives at `build-tools/tools/buck/graph.json`; no change to file path.

## End‑to‑end architecture

High‑level: identical to Go. Buck exports configured graph → zx exporter attaches Rust labels → Nix planner dispatches Rust targets to Rust templates → Nix builds using `buildRustPackage` (or naersk) with per‑crate `cargoPatches` → Providers map labels to Buck dependencies → Only impacted targets rebuild.

Labels for Rust use the existing module form to maximize reuse:

- Format: `module:<crate>@<version>` (crate names have no `/`, so they won’t collide with Go import paths).
- In the current repo, `module:` labels are primarily diagnostic. Provider mapping is driven by `lockfile:` and `nixpkg:` labels in `build-tools/tools/buck/gen-auto-map.ts`.
- Preferred initial wiring: importer‑scoped `lockfile:` labels (e.g., `lockfile:<path/to/Cargo.lock>#<importer>`) so current `gen-auto-map.ts` can map providers without changes. Per‑crate `module:` → provider mapping is optional and would require extending auto‑map.
- Invalidation: regardless of provider model, include package‑ or importer‑local patch files in target `srcs` so Buck invalidation remains precise; provider stamps stay metadata‑only.

## Planner integration

Extend `graph-generator.nix` the same way Go is integrated:

- Dispatch: add a Rust branch to the planner’s registry with predicates similar to `pick` in the Go example. Detection heuristic:
  - Prefer `rule_type` prefix `rust_` or stamped labels `lang:rust` from macros.
  - Kind: `bin` for `rust_binary`/`rust_test` executables, `lib` for libraries.
- Inputs:
  - `Cargo.lock` path for the target’s workspace root (see exporter section for precise scoping).
  - Optional `features`, `target` triple, `release` flag (build configuration knobs forwarded via macros).
- Outputs:
  - `rustTargets` attrset analogous to `goTargets`.
  - Combined `all` output with symlinks (unchanged pattern).

## Nix language template (build-tools/tools/nix/templates/rust.nix)

Implement two functions `rustApp` and `rustLib` consumed by `build-tools/tools/nix/lang-templates.nix`. Keep this file tiny; push helpers into `lang-templates.nix` where possible.

Key design points:

- Use `pkgs.rustPlatform.buildRustPackage` for both app and lib, mirroring Go’s simplification. If later we adopt `naersk`, the interface stays the same; only internals change.
- Patches are applied to vendored crates using `cargoPatches`, with map construction via `H.patchesMapFromDir patchDir` from `build-tools/tools/nix/lib/lang-helpers.nix`.
- Dev overrides are read via `H.readDevOverrides devOverrideEnv` (JSON map of `crate@version` → local path). In Nix, these are converted with `builtins.path` to deterministic store paths and injected during the patch phase to replace the vendored crate directory. CI enforcement uses `H.guardNoDevOverridesInCI devOverrideEnv`.
- Features, target triple, and release flag are threaded through to the builder.

Sketch (illustrative):

```nix
{ pkgs }:
let
  lib = pkgs.lib;
  H = import ../lib/lang-helpers.nix { inherit pkgs; };
in {
  rustApp = { name
            , cargoLock
            , patchDir ? ../../patches/rust
            , devOverrideEnv ? "NIX_RUST_DEV_OVERRIDE_JSON"
            , features ? []
            , release ? true
            , target ? null
            }:
    let
      patchesMap = H.patchesMapFromDir patchDir;
      devOverrides = H.readDevOverrides devOverrideEnv;
      _ = H.guardNoDevOverridesInCI devOverrideEnv;
      cratePatches = lib.concatLists (lib.attrValues patchesMap);
    in pkgs.rustPlatform.buildRustPackage {
      pname = "rust-${name}";
      version = "0.1.0";
      src = ./.;
      cargoLock.lockFile = cargoLock;
      cargoPatches = cratePatches;
      buildFeatures = features;
      CARGO_BUILD_TARGET = if target == null then null else target;
      buildType = if release then "release" else "debug";
      # Replace vendored crates with dev overrides by copying into cargo deps after vendor unpack
      postPatch = lib.optionalString (devOverrides != {}) (
        ''
        if [ -d cargo-vendor ]; then cd cargo-vendor; fi
        for kv in ${builtins.concatStringsSep " " (builtins.attrNames devOverrides)}; do
          crate="${lib.strings.escapeShellArg (builtins.substring 0 200 "${" + "crate" + "}")}" # placeholder to satisfy quoting
        done
        ''
      );
    };

  rustLib = args: rustApp args // { pname = "rustlib-${args.name}"; };
}
```

Notes:

- We rely on `cargoPatches` to apply patches to vendored crates. Patches must be generated with paths relative to the cargo vendor root (the patching workflow below ensures this).
- The `postPatch` override logic is intentionally left schematic in this sketch; the implementation will map each `crate@version` to `vendor/<crate>-<version>` and replace its contents when dev overrides are set. This mirrors the Go pattern (override per module) with the Rust vendor layout.

## Buck macros (build-tools/rust/defs.bzl)

Provide thin wrappers that stamp labels and append providers from `auto_map.bzl`:

- `nix_rust_library(name, srcs, deps = [], features = [], labels = ["lang:rust", "kind:lib"], ...)`
- `nix_rust_binary(name, srcs, deps = [], features = [], labels = ["lang:rust", "kind:bin"], ...)`
- `nix_rust_test(name, srcs, deps = [], features = [], labels = ["lang:rust", "kind:test"], ...)`

Behavior:

- Stamp `lang:rust` and `kind:*` labels.
- Forward configuration that affects dependency resolution to the exporter (features, target triple if provided via attrs/constraints).
- Append provider deps using `realize_provider_edges(...)` (from `//build-tools/lang:defs_common.bzl`) and `MODULE_PROVIDERS` loaded via `//build-tools/lang:auto_map.bzl`.

## Exporter (authoritative labels via cargo metadata)

Extend `build-tools/tools/buck/export-graph.ts` with a Rust adapter:

- Batch by config tuple to minimize `cargo metadata` runs:
  - `(toolchain hash, workspace root, target triple, release/debug, sorted features)`.
- For each batch:
  - Run `cargo metadata --format-version=1 --no-deps=false` with appropriate `--filter-platform <triple>` and feature flags.
  - Build a crate index `{ crateName -> version }` over the reachable graph for each target in that batch.
  - Attach `module:<crate>@<version>` labels to each target that actually reaches those crates (not global to the workspace).
- Cache results by the config tuple + content hash of `Cargo.lock` (and optionally `Cargo.toml`s under the workspace root) to avoid redundant work.
- Tests mirror Go’s: prove that test‑only features/crates label test targets only.

Assumption: `cargo metadata` provides sufficient detail to map feature sets per target; where target‑specific features differ, the batching key ensures correct scoping.

## Provider sync (Rust)

Add `build-tools/tools/buck/sync-providers-rust.ts`:

- Scan `patches/rust/*.patch` (flat). Derive `crate@version` from filenames; enforce “one patch per crate@version”; warn on subdirectories.
- Emit `third_party/providers/TARGETS.rust.auto` with one rule per crate@version:
  - Rule: `rust_crate_patch(name, crate_key, patch_path)` (defined in `//third_party/providers/defs_rust.bzl`).
- Deterministic, idempotent output, sorted entries. Collisions error out with actionable messages.

Provider rule (defs):

```starlark
def rust_crate_patch(name, crate_key, patch_path):
    genrule(
        name = name,
        srcs = [patch_path],
        out = name + ".stamp",
        cmd = "if command -v sha256sum >/dev/null; then cat $SRCS | sha256sum > $OUT; else cat $SRCS | shasum -a 256 > $OUT; fi",
        visibility = ["//visibility:public"],
    )
```

Provider naming: reuse `providerNameForModuleKey(crate, version)` from `build-tools/tools/lib/providers.ts` so `gen-auto-map.ts` needs no changes.

## WASM Targets

Now that repository macros and facilities for WASM are available (freestanding and WASI), Rust will ship with WASM options:

- Target triples: `wasm32-unknown-unknown` (freestanding) and `wasm32-wasi` (WASI).
- Buck macros: extend `build-tools/rust/defs.bzl` with `nix_rust_wasm_library` and `nix_rust_wasi_binary` (or a `wasm = "freestanding"|"wasi"` attr) that:
  - Stamp `lang:rust` and `kind:wasm|wasi` labels.
  - Forward `--target` and features to the builder.
  - Append providers using `MODULE_PROVIDERS` loaded via `//build-tools/lang:auto_map.bzl` and shared provider-edge wiring helpers.
- Planner/templates: add `rustWasmLib`/`rustWasiBin` functions in `build-tools/tools/nix/templates/rust.nix`; compile with cargo for the requested target; reuse patch/override maps.
- Tests: load freestanding `.wasm` via `WebAssembly.instantiate`; run WASI via `node:wasi` and assert exported functions (e.g., `add(2,3)=5`).

Phasing: freestanding first, WASI shortly after; both reuse existing glue, labels, and providers.

## Patching workflow (`patch-pkg rust`)

Add a Rust language handler `build-tools/tools/patch/patch-rust.ts` behind the existing `patch-pkg` CLI interface with the same subcommands and behavior as Go:

- `start <crate>`
  - Resolve the vendor source directory for `<crate>@<version>` using `cargo vendor` (or by reading Nix’s vendored cargo deps path when available).
  - Create a writable temp clone of just that crate’s source.
  - Update `NIX_RUST_DEV_OVERRIDE_JSON` for `<crate>@<version>` to point to the temp path; print a warning that local derivation hashes will differ.
  - Launch `$PATCH_EDITOR` if set.
- `apply <crate>`
  - Generate a unified diff against the vendored crate path so that patch paths are anchored to the vendor root (compatible with `cargoPatches`).
  - Write file: `patches/rust/<crate>@<version>.patch` (flat dir).
  - Run glue: `sync-providers` (Rust driver included) and `gen-auto-map`.
  - Clear the dev override entry and remove the temp dir.
- `reset <crate>`
  - Discard temp dir and remove the dev override entry.
- `session <crate>`
  - Long‑lived session: Ctrl‑C → reset, Ctrl‑D → apply.

Notes:

- Resolution of `<crate>@<version>`: the handler consults the exporter’s cache or runs `cargo metadata` to find the exact version from the workspace `Cargo.lock` used by the selected target(s). When ambiguous, prompt for disambiguation (`crate` may appear multiple times at different versions).
- In CI, `NIX_RUST_DEV_OVERRIDE_JSON` must be unset; the Nix templates throw if overrides are present and `CI=true`.

## Key assumptions to validate

- `cargoPatches` in `buildRustPackage` applies cleanly to vendored crates when patch paths are relative to the vendor root. We will generate patches in that shape.
- `cargo metadata` with appropriate features and target triple provides accurate crate/versions for per‑target labeling.
- Mapping `crate@version` → `vendor/<crate>-<version>` is stable across the chosen vendor workflow (either cargo vendor or Nix’s cargo vendor unpack path). If using `naersk`, we’ll confirm equivalent paths or adapt patch anchoring.
- `NIX_RUST_DEV_OVERRIDE_JSON` implemented with `builtins.path` is acceptable for local builds and appropriately forbidden in CI (hash changes are expected locally and are warned).

## Risks and mitigations

- Patch anchoring mismatch (risk: patches don’t apply):
  - Mitigation: enforce patch generation against the exact vendor layout used in Nix (`patch-rust.ts` uses the same vendor path), add a smoke test that applies a sample patch and builds.
- Feature/target drift between exporter and build (risk: labels don’t match build reality):
  - Mitigation: exporter batches by the same config tuple the macros forward; add tests flipping features/targets and assert correct label changes.
- Dev overrides break purity (risk: impure inputs leak into CI):
  - Mitigation: guard in Nix (`CI=true` throws) and in CLI; provide `build-tools/tools/dev/clear-overrides.ts` to reset environment.
- Multiple versions of the same crate (risk: ambiguous `start crate`):
  - Mitigation: CLI prompts to choose `crate@version` when more than one is found in `Cargo.lock`.
- Ecosystem choice (naersk vs buildRustPackage vs crate2nix):
  - Mitigation: template API is stable; internal implementation can swap without affecting planner/exporter/patch UX.

## Areas of concern

- Per‑crate dev overrides: implementing a clean override of a vendored dependency requires careful path mapping. We will start with patch‑only (Phase 1) and add dev overrides (Phase 2) to reduce complexity.
- Workspace graphs with multiple `Cargo.lock`s: initial exporter support assumes one workspace root per Rust target group; we’ll detect and handle multiple roots incrementally.
- Non‑crates.io sources (git/path): labeling uses the resolved version from `Cargo.lock`. For git sources, the effective version has a pseudo‑version or rev; we will encode `<crate>@<rev>` and expect patches to apply to that snapshot.

## Phases and acceptance

Phase 1 — Minimal viable Rust integration

- Add `build-tools/tools/nix/templates/rust.nix` with `rustApp`/`rustLib` and `cargoPatches` support; no dev overrides.
- Add `build-tools/rust/defs.bzl` macros to stamp labels and append providers.
- Add `build-tools/tools/buck/sync-providers-rust.ts` and `third_party/providers/defs_rust.bzl`.
- Extend exporter to label Rust targets via `cargo metadata`.
- Verify with a tiny sample crate (lib + bin). Glue runs end‑to‑end; provider updates cause only affected targets to rebuild.

Phase 2 — Dev overrides and hardening

- Implement `NIX_RUST_DEV_OVERRIDE_JSON` handling in the Nix template (`postPatch` that replaces vendored crates with `builtins.path` inputs).
- Add exporter caching, multi‑root handling, and feature matrix tests.
- Add e2e smoke test changing an unrelated patch to confirm no rebuild for unrelated targets.

Completion criteria

- Rust targets build under Buck with providers and auto‑map wiring in place.
- Changing a crate patch invalidates only targets that reach that crate.
- CI guards pass; dev overrides forbidden in CI; local warnings printed when overrides are set.
- Tests for provider determinism, exporter correctness, and wiring are green.
