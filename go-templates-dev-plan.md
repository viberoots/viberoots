## Goal

Create production-ready Copier templates in `tools/scaffolding/templates/go/*` to scaffold a Go monorepo compatible with our Nix + Buck2 + dynamic-derivations build system and patching workflow, and make them usable in e2e tests.

## Scope and completion criteria

- **Scope**: Flesh out `go/cli` and `go/lib` templates; add a top-level monorepo seed where appropriate; wire generated projects to Buck2 macros, provider auto-map, and Nix planner conventions; integrate test and lint skeletons; minimal docs.
- **Done when**:
  - `scaf new go lib <name>` produces a buildable, testable Go library wired into Buck2 macros and our provider mapping.
  - `scaf new go cli <name>` produces a buildable Go CLI, imports a local lib by example, and runs under Buck2.
  - Generated projects align with `build-system-design.md` (graph export, provider sync, auto-map, prebuild-guard) and pass validation with `tools/scaffolding/validate.ts`.
  - E2E scaffold tests verify build and minimal test execution using Buck2.
  - Presence of a scaffold directory with its own `TARGETS` (using `nix_go_*` macros) is sufficient for Buck2 discovery via `//...` — no central registry.
  - HARD REQUIREMENT (partial clones): Decentralized registration is mandatory so that a partial clone containing only certain `apps/<name>` or `libs/<name>` directories (plus required shared files) can be built without modifying any central registry.

## Components and dependencies

- **Buck2 integration**: `//go:defs.bzl` macros (`nix_go_library`, `nix_go_binary`, `nix_go_test`) must exist; `third_party/providers/auto_map.bzl` is generated by our tools; `prebuild-guard.ts` enforces presence.
- **Nix integration**: `tools/nix/lang-templates.nix` provides Go derivations; planner `graph-generator.nix` consumes `tools/buck/graph.json`.
- **Patching**: `patches/go/` flat directory present; provider sync and auto-map are invoked by CI or `patch-pkg apply`.
- **Scaffolding CLI**: `scaf new` uses Copier; templates must supply `copier.yaml`, `meta.json`, files, and README.md.jinja.
- **Execution environment (required)**: All tasks and tests must run with the repo’s direnv-loaded dev shell so required tools (buck2, nix, go, pnpm, zx) are on PATH. Do not manually modify PATH inside tests; rely on the shell environment.

## Decentralized registration (design intent and requirement)

- **Hard requirement (supports partial clones)**: Each scaffolded package owns its `TARGETS` file under `libs/<name>/` or `apps/<name>/`, using `load("//go:defs.bzl", "nix_go_library", "nix_go_test", "nix_go_binary")` and declaring its targets. Buck2 discovers these with `//...`; no central, shared registry or aggregator files may be required for new packages.
- **Prohibited**:
  - Central lists of packages, top-level aggregators that must be edited per instantiation, or any manual provider wiring in package `TARGETS`.
- **Touch points**:
  - Per-package `TARGETS` exists and uses `nix_go_*` macros (providers are attached via generated `auto_map.bzl`).
  - After adding or changing packages, run glue steps (or let CI run them): `export-graph.ts`, `sync-providers.ts`, `gen-auto-map.ts`; `prebuild-guard.ts` ensures freshness.
  - After dependency changes in a package, run `tools/dev/install-deps.ts` so `gomod2nix.toml` is regenerated.
  - Only if introducing custom rule names that do not resolve to `go_*`, add a mapping entry in `tools/nix/mapping.nix` to route to the Go template (not required for `nix_go_*` macros wrapping `go_*`).
  - Ensure `//go/defs.bzl` is present in any partial clone you want to build; `tools/scaffolding/resolver.json` only controls destination paths, not build registration.
- **Acceptance**:
  - Creating a new `libs/<name>` or `apps/<name>` directory with a valid `TARGETS` and running glue steps is sufficient for `buck2 build //...` to include it automatically.
  - Partial clone smoke: In a temp dir, copy only `go/defs.bzl`, `third_party/providers/auto_map.bzl` (or regenerate glue), minimal Buck config, and a single scaffolded package; `buck2 build //...` succeeds without editing any central registry.

## Phases

### Phase 1 — Template metadata and variables

- Add/verify `meta.json` fields for `go/cli` and `go/lib`:
  - `language`, `template`, `description`.
  - `help.usage`, `help.notes`, `help.examples`.
- Define Copier inputs in `copier.yaml` for both templates:
  - `name` (string), `module` (e.g., `github.com/org/{{ name }}`), `description`.
  - Optional: `go_min` (default 1.22), `license` (SPDX id), `enable_ci` (bool).
- Acceptance: `scaf validate all` passes.

### Phase 2 — Directory layout and essential files

- `go/lib` template layout:
  - `libs/{{ name }}/`
    - `go.mod` (module = `{{ module }}`), `go.sum` (empty allowed), `gomod2nix.toml` placeholder.
    - `TARGETS` with `nix_go_library` and `nix_go_test` using our macros.
    - `pkg/{{ name }}/{{ name }}.go` minimal API.
    - `pkg/{{ name }}/{{ name }}_test.go` minimal single-test file.
    - `README.md` from `README.md.jinja`.
- `go/cli` template layout:
  - `apps/{{ name }}/`
    - `go.mod` (module = `{{ module }}`), `gomod2nix.toml` placeholder.
    - `TARGETS` with `nix_go_binary` and example dep on a local lib via comment.
    - `cmd/{{ name }}/main.go` hello-world using a function shape that mirrors lib usage.
    - `README.md`.
- Acceptance:
  - Files render using Copier with `scaf new go lib demo-lib` and `scaf new go cli demo-cli`.

### Phase 3 — Buck2 wiring and provider auto-map

- In each generated `TARGETS`:
  - Use `labels = ["lang:go", "kind:lib|bin"]` on raw `go_*` rules if any are present; for macros, rely on `go/defs.bzl`.
  - Ensure macros used are `nix_go_*`; do not directly depend on providers; providers come via `auto_map.bzl`.
- Add a repo-level guard call guidance in the generated README: run `node tools/buck/export-graph.ts`, `node tools/buck/sync-providers.ts`, `node tools/buck/gen-auto-map.ts`, then `buck2 build`.
- Acceptance: After creating templates in this repo, a sample scaffold builds via `buck2 build //apps/demo-cli:demo-cli` and `buck2 test //libs/demo-lib:demo-lib_test` once glue is generated.

### Phase 4 — Go module and gomod2nix bootstrapping

- In `go.mod` skeletons set `go {{ go_min }}` and example `require` lines only when needed.
- Add post-step hints in README to run `node tools/dev/install-deps.ts` which should trigger `gomod2nix` integration (as per design).
- Acceptance: Running our install script updates `gomod2nix.toml` deterministically; no manual edits required.

### Phase 5 — Testing conventions

- Each test file contains exactly one test to follow repo test conventions.
- Provide a minimal `_test.go` that compiles and passes; demonstrate `require` usage gated behind a commented import until `go mod tidy` runs.
- Acceptance: `buck2 test` succeeds on generated test targets.

### Phase 6 — Template docs and usability

- `README.md.jinja` should cover:
  - How to build with Buck2 and our glue scripts.
  - How to add a dependency and refresh `gomod2nix.toml`.
  - How to create and apply a Go patch using `patch-pkg`.
  - Where files live and why (`apps/`, `libs/`, `patches/go/`).
- Acceptance: Readme renders with variables and is accurate.

### Phase 7 — E2E scaffold tests (repo-level)

- Add zx tests under `tools/tests/scaffolding/`:
  - `go-lib.scaffold-and-build.test.ts`: create lib, run glue, build, run tests.
  - `go-cli.scaffold-and-build.test.ts`: create cli, run glue, build.
  - `partial-clone.discover-and-build.test.ts`: simulate a sparse/partial clone by rsyncing only `go/defs.bzl`, `third_party/providers/*` (or re-run glue), minimal Buck config, and one scaffolded package; verify `buck2 build //...` discovers and builds it without central updates.
- Test execution model (required):
  - Run each test in a temporary copy of the repo created via rsync, excluding heavy or generated dirs (e.g., `.git`, `node_modules`, `coverage`, `.clinic`, `buck-out`).
  - Rely on direnv to load the dev shell within the temp directory; do not modify PATH manually.
  - Apply project test rules: single test per file, external timeouts, zx wrapper conventions.
- Use the existing zx test runner conventions and Buck2 targets.
- Acceptance: All three tests pass locally and under CI; coverage aggregated if enabled. Tests verify they ran in a temp copy with direnv-provided PATH (no manual PATH edits).

### Phase 8 — Validation and guardrails

- Extend `tools/scaffolding/validate.ts` only if necessary to catch common template errors (missing `copier.yaml`, missing `TARGETS`, etc.). Keep current checks otherwise.
- Add simple lint to flag accidental subdirectories under `patches/go/` and duplicate patch names; wire into `tools/dev/patches-lint.ts` if not already enforced.
- Acceptance: `scaf validate all` clean; lint passes on fresh scaffolds.

### Phase 9 — Monorepo seed (optional but recommended)

- Provide a `go/mono` template to initialize a basic monorepo skeleton:
  - Top-level `TARGETS` placeholder if needed.
  - Create empty `patches/go/`, `third_party/providers/` (empty; auto files are generated), `.gitignore` entries for generated glue.
  - A sample `libs/example-lib` and `apps/example-cli` created via sub-calls to Copier or inline files to prove end-to-end flow.
- Acceptance: `scaf new go mono <name>` yields a repo that builds after running glue steps.

## Systematic tasks with checkpoints

- Template meta and variables
  - Output: Updated `meta.json`, `copier.yaml` for `go/lib` and `go/cli`.
  - Check: `scaf validate all` succeeds.
- Layout and files
  - Output: `apps/{{name}}` and/or `libs/{{name}}` tree with minimal Go code and Buck `TARGETS` using macros.
  - Check: Copier runs successfully; files present.
- Buck2 wiring
  - Output: Valid `TARGETS` that build once glue is generated.
  - Check: Build selected scaffold targets.
- gomod2nix integration
  - Output: `gomod2nix.toml` generated/updated.
  - Check: Install script runs without errors; subsequent builds succeed.
- Tests
  - Output: Minimal passing tests per target, one-test-per-file.
  - Check: `buck2 test` green.
- Docs
  - Output: README per scaffold with accurate steps.
  - Check: Manual read-through and smoke run.
- E2E
  - Output: zx tests that exercise scaffold → glue → build/test.
  - Check: Tests pass locally and CI.

## Risks and mitigations

- Glue file freshness
  - Mitigation: In docs, clearly instruct to run export-graph, provider sync, auto-map before first build; rely on `prebuild-guard.ts`.
- gomod2nix drift
  - Mitigation: Emphasize using `tools/dev/install-deps.ts`; avoid manual edits.
- Provider name collisions
  - Mitigation: Use shared `tools/lib/providers.ts` helpers in generators; tests cover wiring.
- Partial clone missing shared files
  - Mitigation: Document the minimal set required in partial clones (`go/defs.bzl`, Buck config, optionally regenerated glue). The `go/mono` seed should place these in stable paths so sparse checkouts can include them predictably.
