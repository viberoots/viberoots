## Nix‑Built C++ Tests — Design, Plan, and Acceptance Criteria

This document specifies how we will run C++ unit tests (gtest) by building the test binaries via Nix and invoking them from Buck using an external runner. The goal is to align with our core design principles: Buck orchestrates “what,” Nix decides “how,” and glue is zx and not committed.

### Problem Statement

Today, Buck’s native `cxx_test` compiles test sources and must know include/link flags. Our philosophy is to keep Buck identity‑only for third‑party deps and push toolchain flags to Nix. This creates a gap for C++ tests: identity‑only providers (e.g., `nixpkg:pkgs.gtest`) don’t supply include/link flags to Buck, so `#include <gtest/gtest.h>` fails unless we add a local shim. The shim violates the “no local prebuilts” policy and leaks implementation details into Buck.

### Objectives

- Eliminate the local gtest shim and hardcoded flags in Buck.
- Keep providers identity‑only (labels like `nixpkg:pkgs.gtest`).
- Build and link C++ test binaries via Nix using the same templates that inject includes/ldflags.
- Run the resulting Nix‑built binaries under Buck tests with coverage parity and deterministic inputs.

### Non‑Goals

- We will not add include/link flags to Buck providers.
- We will not depend on vendored third‑party C++ headers/libs.

---

## High‑Level Architecture

- Buck macros under `//cpp/defs.bzl` gain a new rule: `nix_cpp_test`.
- `nix_cpp_test` mirrors `nix_cpp_binary`/`nix_cpp_library` but produces a test runnable by Buck via `ExternalRunnerTestInfo`.
- The test binary itself is built by calling into Nix (via our dynamic derivation planner/templates) using the same `nixCxxAttrs` derived from provider labels `nixpkg:*` in the test’s transitive deps.
- Buck wires identity‑only providers into the target graph so changes to patch overlays or nixpkgs pins invalidate the right test targets.

Diagram:

- Buck (macro) → Nix planner/templates → build test binary
- Buck external runner → invokes the built binary with gtest flags

## Detailed Design

### 1) Planner and Template Inputs (Nix)

- Reuse the existing `build-tools/tools/nix/planner/cpp.nix` plugin:
  - `isTarget(n)` detects `cxx_*` or `lang:cpp`.
  - `kindOf(n)` is `bin|lib|test` from `rule_type` or labels.
  - `collectNixAttrsFor(name)` DFS over deps to collect `nixpkg:*` labels → `nixCxxAttrs`.
  - New function for tests: `mkTest(name)` which delegates to a new template function `T.cppTest`.

- Add `cppTest` to `build-tools/tools/nix/templates/cpp.nix`:
  - Inputs: `name`, `srcRoot`, `subdir`, `includes`, `defines`, `cflags`, `ldflags`, `std`, `nixCxxAttrs`.
  - Resolve `nixCxxAttrs` to nixpkgs values; inject `-isystem <pkg>/include` per attr. If we later need lib flags, add deterministic `-L<lib>` and `-l<name>` (sorted, stable).
  - Build a single test binary `out/bin/<sanitized-name>` containing all test objects.
  - Determinism: sorted source discovery, stable flag ordering, `-ffile-prefix-map`, no ambient FS reads.

### 2) Buck Macro: `nix_cpp_test` (Starlark)

- File: `cpp/defs.bzl`.
- Behavior:
  - Accepts similar attrs to `nix_cpp_binary` with `srcs`, `deps`, optional `includes`/`defines`/`cflags`/`ldflags`.
  - Stamps `lang:cpp` + `kind:test` using `stamp_labels` from `lang/defs_common.bzl`.
  - Appends identity‑only providers from `MODULE_PROVIDERS` (generated by `gen-auto-map.ts`).
  - Generates a tiny “runner” wrapper using `ExternalRunnerTestInfo` that:
    - Resolves the Nix output path for this target (via our graph‑generator package outputs, same approach as for binaries).
    - Executes the built test binary with environment suitable for coverage.
    - Emits a no‑op stamp file as the Buck output to satisfy Buck’s expectations.

Notes:

- The macro does not compile C++; it only orchestrates inputs and declares the external runner step.

### 3) Buck↔Nix Output Wiring

- We already expose per‑target derivations through `graph-generator` outputs in the flake under `packages.<system>.graph-generator`. Extend it to include C++ tests:
  - Add `cppTargets` similar to Go (names → derivation paths).
  - The macro records its label → the external runner loads the path from flake outputs using an environment variable (or reads a small manifest produced by `graph-generator`).

### 4) Providers and Auto Map

- No change in policy: providers are identity‑only rules with `labels = ["lang:cpp", "nixpkg:pkgs.<attr>"]`.
- `build-tools/tools/buck/gen-auto-map.ts` already maps `nixpkg:*` labels to named provider targets for wiring.
  - Tests include those providers via auto map or explicit deps; changing nixpkgs attr or overlays invalidates only impacted tests.

### 5) Coverage & Test Runner Semantics

- For C++ tests we run the test binary directly (no Node coverage); Buck still aggregates test results and logs under `buck-out/test-logs`.

## Implementation Plan (Engineer‑Ready)

1. Nix Templates
   - Add `cppTest` to `build-tools/tools/nix/templates/cpp.nix` mirroring `cppApp` with deterministic compile/link and `nixCxxAttrs` resolution.
   - Ensure headers are copied to `$out/include` for diagnostics; binary to `$out/bin/<name>`.

2. Planner Plugin
   - In `build-tools/tools/nix/planner/cpp.nix`, add `mkTest(name)` using `T.cppTest { nixCxxAttrs = collectNixAttrsFor name; ... }`.
   - Export tests in a new attr set (e.g., `cppTargets.tests`) for `graph-generator` outputs.

3. Graph Generator
   - Extend `build-tools/tools/nix/graph-generator.nix` to include C++ targets (bin, lib, test) in the outputs attrset, similar to how Go is handled.
   - Keep partial‑clone safe and avoid heavy repo scans.

4. Buck Macros
   - Add `nix_cpp_test` to `cpp/defs.bzl`:
     - Stamp labels, append providers, and produce `ExternalRunnerTestInfo` with a tiny shell wrapper that invokes the Nix‑built test binary path for this target.
     - Output a small `.stamp` as Buck output.

5. Remove Shim and Update Scaffolds
   - Delete or neutralize `//third_party/cpp` shim.
   - Update C++ scaffolding templates to use `nix_cpp_test` and depend only on providers.

6. Tests
   - Add a zx test under `build-tools/tools/tests/cpp/` that scaffolds a C++ lib/app + `nix_cpp_test` and runs it; assert no references to the local shim; verify pass.
   - Add a provider wiring test ensuring `nixpkg:*` labels map correctly to providers for test targets.

7. CI
   - Ensure stages remain: export graph → sync providers → gen auto map → prebuild guard → build & test. No glue is committed.

## Acceptance Criteria

- `nix_cpp_test` exists and is documented in `cpp/defs.bzl` comments.
- A target defined with `nix_cpp_test` that uses `#include <gtest/gtest.h>` builds and runs successfully without any local shim.
- Provider wiring: changing a `nixpkg` overlay or label invalidates only impacted C++ tests.
- Full test suite passes locally with coverage enabled per project policy.
- Scaffolding templates for C++ use `nix_cpp_test` by default and pass on first run.

---

## Risks and Mitigations

- Toolchain ABI alignment: Ensure Nix C++ toolchain used in templates matches Buck toolchain expectations when Buck invokes the test binary (typically benign, as Buck only runs the binary). Document expected runtime deps.
- Flake outputs lookup: If graph‑generator outputs are missing or stale, prebuild guard should fail early with a clear message.
- Cross‑platform differences (Darwin/Linux): Keep flags minimal and sorted; add platform conditionals only where necessary and covered by tests.

---

## Consequences of Not Implementing

- Continued reliance on a local shim (`third_party/cpp`) to provide headers, violating our policy and creating drift.
- Mixing “how” (include flags) into Buck rules, increasing duplication and risk of inconsistencies vs. Nix.
- Harder to reason about invalidation: Buck would not reflect changes to Nix overlays or nixpkgs pins through identity‑only providers as cleanly for tests.

- This work touches: `build-tools/tools/nix/templates/cpp.nix`, `build-tools/tools/nix/planner/cpp.nix`, `build-tools/tools/nix/graph-generator.nix`, `cpp/defs.bzl`, scaffolding templates under `build-tools/tools/scaffolding/templates/cpp/*`, and tests under `build-tools/tools/tests/cpp/`.
- Keep functions small (≤10–15 lines), early‑return style, explicitly typed where applicable. Maintain determinism and stable sorts throughout. Avoid ambient FS reads.
- Follow the CI stage order; do not commit glue. Ensure prebuild guard covers the new outputs.
