## PNPM Exporter Adapter — Validate‑Only + Sidecar Index

This document proposes a minimal Node/PNPM exporter adapter that increases uniformity and diagnostics for graph/tooling consumers while preserving a single source of truth for importer‑scoped labels and invalidation behavior.

### Goals

- Consistent, exporter‑level diagnostics for Node targets (like Go/C++ exporters), without changing behavior.
- Uniform, easy‑to‑consume view for tooling by emitting a small, derived sidecar index alongside `graph.json` (generated by glue).
- Keep macros as the single authority for Node importer labels and patch invalidation.

### Non‑Goals

- Do not add or mutate labels in `build-tools/tools/buck/graph.json` for Node targets.
- Do not change invalidation mechanics, provider rules, or macro semantics.
- Do not replace macro‑stamped labels; the adapter is strictly validation + indexing.

## Single Source of Truth (SST)

- Node importer‑scoped labels are stamped by macros (authority):
  - Exactly one `lockfile:<path>#<importer>` per Node target.
  - Importer‑local patches are included in target `srcs` to achieve precise invalidation for patches.
- Provider generation and auto‑mapping already work end‑to‑end; the adapter must not alter this.

## Adapter Scope and Responsibilities (Validate‑Only)

The Node exporter adapter runs during graph export and:

- Detects Node targets by labels (e.g., `lang:node`) or by explicit rule families we own.
- Validates that each Node target has exactly one importer‑scoped lockfile label (`lockfile:<path>#<importer>`):
  - Missing label → finding (warn local, error in CI), with remediation guidance.
  - Multiple labels → finding (warn local, error in CI).
  - Malformed importer (e.g., path/importer mismatch) → finding.
- Emits validation findings only; the sidecar index is generated by glue for tooling uniformity.

Important: the adapter never writes labels back into `graph.json` and never attempts to infer importers. It only reads existing labels and validates them.

## Sidecar Index (Derived, Read‑Only)

Emit a small index next to the graph that mirrors the macro‑stamped importer labels:

- Path: `build-tools/tools/buck/node-lock-index.json` (generated by glue `build-tools/tools/buck/gen-provider-index.ts`).
- Content: mapping of target → importer‑scoped lockfile key.

Example (illustrative):

```json
{
  "//apps/web:bundle": "lockfile:apps/web/pnpm-lock.yaml#apps/web",
  "//libs/ui:build": "lockfile:libs/ui/pnpm-lock.yaml#libs/ui"
}
```

Design notes:

- Built entirely from labels already present in `graph.json` (SST honored).
- If a target lacks a valid label, the adapter emits a validation finding and omits the target from the index.
- Keys are fully‑qualified Buck target labels (`//pkg:rule`); values are lowercase lockfile labels.
- Deterministic and idempotent: entries must be sorted by target label and written via the shared `writeIfChanged` helper.

## Composite Graph API (Single Consumption Point)

Introduce a tiny library and optional CLI so all downstream tools consume a composite view rather than raw files:

- Library: `build-tools/tools/lib/graph-view.ts` (or extend `build-tools/tools/lib/graph.ts`).
- Responsibility:
  - Load `build-tools/tools/buck/graph.json` (exported graph).
- Load sidecars: `third_party/providers/provider_index.json` (already generated) and `build-tools/tools/buck/node-lock-index.json` (generated by glue).
  - Return a normalized structure that includes per‑target provider mapping (Node lockfile providers, nixpkg providers, and gated Go module providers) and the importer lock mapping.
- CLI: `build-tools/tools/buck/graph-view.ts` prints the composite view to stdout for scripts/dashboards.

See also:

- Handbook conventions for the Composite Graph API and sidecar expectations: `docs/handbook/conventions.md` (Graph consumption section).

Contract:

- All internal tooling must consume the Composite Graph API; direct reads of `graph.json` are unsupported.
- Provide a single exported TypeScript type (e.g., `CompositeGraphView`) so consumers share one data contract.

Benefits:

- Uniformity for graph/tooling consumers with zero change to build behavior.
- Future tooling reads one API; we can evolve internals without breaking consumers.

## CI/Lint Mitigations (Prevent Raw graph.json Consumption)

- Add a lightweight CI “tooling‑contract” check:
  - Forbid new scripts from reading `build-tools/tools/buck/graph.json` directly; allowlist the composite API and exporter internals.
  - Simple grep/checkstyle gate with exceptions for known code paths.
- Optional ESLint custom rule for repo scripts/packages that flags direct reads of `graph.json`.
- Exporter severity: `warn` locally, forced to `error` in CI when Node label validation fails.
- Prebuild guard: verify presence/freshness of sidecars (just like `auto_map.bzl`). Fail when `build-tools/tools/buck/node-lock-index.json` is missing or older than `build-tools/tools/buck/graph.json`. Optionally, also fail when it is older than any `TARGETS`/`*.bzl` inputs that affect labeling.

## Schema and Versioning

- Add `$schema` and `version` fields to both `build-tools/tools/buck/graph.json` and `build-tools/tools/buck/node-lock-index.json`.
- The exporter prints a brief note pointing consumers to the Composite Graph API and the schema version on successful export.
- Document the schema URL and versioning policy in `docs/handbook/conventions.md`.

## Developer Experience (DX)

- App developers: unchanged. They keep using macros that stamp importer labels and include importer‑local patches. Patching with `patch-pkg` works the same.
- Tooling/ops: gain a stable, uniform data source (composite view) and clear diagnostics for Node label issues.

## Risks and Mitigations

- Risk: Future tools ignore the composite API and parse `graph.json` directly.
  - Mitigation: CI gate + ESLint rule; link to the composite API in exporter logs and docs; add a `"$schema"` + banner to graph output pointing to the composite API.
- Risk: Sidecar drift.
  - Mitigation: Generate sidecar during glue (`gen-provider-index.ts`); prebuild guard enforces freshness and auto-fixes locally.
- Risk: Duplicate semantics.
  - Mitigation: Adapter is validate‑only; no label synthesis; sidecar is a projection from existing labels.

## Rollout Plan

1. Implement the adapter (validate‑only):
   - Add `build-tools/tools/buck/exporter/lang/node.ts` with `adapter: { name: "node", isNode, validate, attachLabels: pass‑through }`.
   - `validate` checks Node targets for exactly one importer label; add findings.
   - Sidecar emission is handled by glue (`build-tools/tools/buck/gen-provider-index.ts`) reading `graph.json`.
   - Ensure sidecar emission is deterministic (sorted keys) and uses `writeIfChanged`.

2. Composite API + CLI:
   - Add `build-tools/tools/lib/graph-view.ts` to merge `graph.json` + provider/lock sidecars.
   - Add `build-tools/tools/buck/graph-view.ts` CLI for scripts.
   - Define and export a single TypeScript type for the composite view consumed by all tools.

3. Policy & enforcement:
   - CI “tooling‑contract” check; optional ESLint rule.
   - Docs update: reference composite API in `docs/handbook/conventions.md` and exporter logs.
   - Extend prebuild guard to check sidecar freshness vs `graph.json` (and optionally `TARGETS`/`*.bzl`).
   - Add a CI grep gate to forbid new raw `graph.json` reads (allowlist exporter internals and the composite API).
   - Add CODEOWNERS coverage for `build-tools/tools/buck/**` and `build-tools/tools/lib/graph-view.ts`; PR template checkbox for Composite API usage.

4. Tests:
   - Adapter validation tests: missing/multiple/malformed lockfile labels → warn locally, error in CI.
   - Sidecar determinism: stable sort and `writeIfChanged` idempotency.
   - Prebuild-guard: fail when sidecar is missing or stale relative to `graph.json` (and optionally `TARGETS`/`*.bzl`).

## Appendix A — Sidecar JSON Schema (Draft)

```json
{
  "$schema": "https://example.com/schemas/node-lock-index.schema.json",
  "title": "Node Lock Index",
  "type": "object",
  "additionalProperties": {
    "type": "string",
    "pattern": "^lockfile:.+#[^#]+$"
  },
  "description": "Map of Buck targets to importer-scoped PNPM lockfile labels."
}
```

## Appendix B — Consumer Sketch (Composite API)

```ts
// build-tools/tools/lib/graph-view.ts (sketch)
import { readGraph } from "./graph";
import fs from "fs-extra";

export async function readCompositeGraph() {
  const nodes = await readGraph("build-tools/tools/buck/graph.json");
  const provIdx = await fs.readJson("third_party/providers/provider_index.json").catch(() => ({}));
  const nodeLockIdx = await fs
    .readJson("build-tools/tools/buck/node-lock-index.json")
    .catch(() => ({}));
  return { nodes, providerIndex: provIdx, nodeLockIndex: nodeLockIdx };
}
```

This design delivers exporter‑level uniformity and diagnostics while preserving a single source of truth in macros and keeping the developer experience unchanged.
